​​​​​​Overview

This addon is used to help simulate ECUs on a CAN network (restbus simulation). More specifically, it generates frames with protection values (CRC or checksums) and keep alive counters (or rolling counters).

Requirements

These drivers must be installed on the deployment computer/target to function.

  • NI VeriStand 2019 or later
  • NI XNET 19.0 or later
  • Network Streams 19.0 or later​

Restbus Simulation Basics

Most automotive ECUs reside on a CAN network. When testing a single ECU, the rest of the network needs to be present to make that ECU happy (operate normally). Instead of having all the physicial ECUs wired up, simulate their CAN traffic on the network. For most of the CAN traffic, NI XNET hardware and NI VeriStand can be used to do this out of box. Once configured, NI VeriStand will generate the simulated messages. Using models, manual UI objects, or automation, the user can set the CAN signals to valid ranges that make the ECU operate normally.​

CAN Signal Protections

While this works for most CAN signals, there is one area that takes additional steps. Some CAN signals are important enough that the ECU manufacturers put protections on them to make sure everything is connected and working. The most common types of protection are rolling counters and checksums/CRCs.​

Rolling Counter

The most basic form of protection is to send a rolling count along with the frame being protected. Essentially, a counter signal is incremented by one and included in the frame each time the frame is sent. There is some upper value at which the count starts over. This is done by the generating ECU, or in the simulation case, the software simulating the generating ECU. The receiving ECU simply reads the frame and makes sure the rolling count incremented in an expected fashion. If the ECU notices a problem, it will generate a diagnostic trouble code (DTC).​

Checksum/CRC Protection

Another form of protection involves an algorithm to calculate a signal value. This calculation happens on the raw value(s) of the signal(s) being protected. The rolling count is often included in the calculation as well. The resulting calculated value is placed into the same frame as the signal(s) being protected. This is done on the generating side of the frame. The receiving side does the same calculation upon reading in the frame. It then checks its locally calculated value with the calculated value received in the frame. If they don't match, the ECU generates a diagnostic trouble code (DTC). The most common algorithms used are Checksum and CRC calculations.​​

Rolling counter and CRCs in NI VeriStand

There are built-in ways of doing rolling counters and CRCs in NI VeriStand (See Configuring CRCs and Counters for Outgoing NI-XNET Frames). If these options work for the ECU under test, use them. They execute on the NI-XNET hardware and therefore consume no CPU resources. If the built in options don't work, use the PROVEtech:RBS by AKKA or the Protection and Counter Addon. ​The RBS addon is best suited for this type of application and is the recommended approach. If the user wants to own the source code, the Protection and Counter Addon addon can work but has less features.

Addon Specifics​

Before using this addon, add the standard frames (without counters or protections) to the standard NI-XNET section.

Addon Configuration​

After setting up all the standard frames, add the protection and counter addon. Upon adding the addon, a configuration GUI will open. Select the desired CAN database and base baud rate.

​

T​he addon will be added to the system definition. Configure the CAN settings on the main page.

Expand the Outgoing section and click on the desired frame type (Cyclic, Event Triggered, Cyclic Event). Select the Import Frames button. Only import frames that contain protection or counter signals. Again, standard frames should be in the XNET section as discussed above.

Notice the built in help. This help provides the details needed to use most sections.

After importing, configure the frame settings. These will be similar to the standard CAN frame settings.

Next, right click on the signals that are either protections or counters. Select the desired type.

​Configure the signal. The counter signal configuration requires a max count value.

 

For protections, first select the algorithm.

 

If the required protection algorithm is not included, it is not supported. A LabVIEW developer could add it to the source code in Frame Protection SubModule and rebuild the addon. Better would be to check for support inPROVEtech:RBS by AKKA and use that.

If the required protection algorithm is supported, select the signals needed for the calculation.

​It is important to note that there are also Status channels for monitoring the addon performance.
